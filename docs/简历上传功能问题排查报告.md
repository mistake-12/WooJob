## 二、代码层问题分析

### 2.1 Storage 上传 — 根因最可能在 Session 传递

| 组件 | 文件 | 问题 |
|------|------|------|
| 上传入口 | `BottomShelf.tsx:271` | `createBrowserSupabaseClient()` 从浏览器 cookie 读取 session |
| Cookie 来源 | `app/auth/callback/route.ts` | Server 端 exchangeCodeForSession 后写入 cookie |
| 中间件 | `proxy.ts` | 仅保护 `/` 和 `/login` 路由，不拦截 Storage 上传（无影响） |

**分析：**

1. `createBrowserSupabaseClient()` 内部调用 `createBrowserClient`，它从 `document.cookie` 读取 Supabase session
2. 登录成功后，`/auth/callback` 通过 `response.cookies.set()` 写入 cookie
3. 写入成功 → 浏览器再次请求时，`createBrowserSupabaseClient()` 读到 cookie → session 有效 → RLS 通过
4. **写入失败或读取失败** → session 为空 → RLS 拒绝 → 报错

### 2.2 潜在代码问题

#### 问题 1：`/auth/callback` Cookie 写入时缺少 SameSite 属性（最可疑）

`route.ts:28-31` 当前写法：

```typescript
setAll(cookiesToSet) {
  cookiesToSet.forEach(({ name, value, options }) => {
    request.cookies.set(name, value);
    response.cookies.set(name, value, options);  // ← options 来自 Supabase SDK
  });
}
```

Supabase SDK 默认的 cookie options 通常没有显式设置 `SameSite`，在 Vercel 跨域场景下，浏览器可能不保存 cookie，导致 `createBrowserSupabaseClient()` 始终读不到 session。

#### 问题 2：`createBrowserSupabaseClient()` 每次调用都新建实例

`BottomShelf.tsx:271, 319` 在 `uploadWithRetry` 和 `handleUpload` 中各自调用 `createBrowserSupabaseClient()`，每个新实例都会重新初始化，可能导致 session 状态丢失。建议复用单一实例。

#### 问题 3：上传失败后 `supabase.storage.from('resumes').remove()` 可能也失败

`BottomShelf.tsx:336` 的删除调用同样依赖 session，如果 session 无效，这个删除也会失败并抛出未捕获错误。

## 三、Supabase 配置检查清单

### 3.1 Storage Bucket `resumes` 配置

- [ ] **Bucket 存在**：Supabase Dashboard → Storage → `resumes` bucket
- [ ] **Public 状态**：如果文件需要公开访问，bucket 设为 Public；如果需要认证访问，设为 Private
- [ ] **Allowed Extensions**：限制为 `.pdf`（如已配置）
- [ ] **File Size Limit**：10MB（与代码 `MAX_FILE_SIZE = 10 * 1024 * 1024` 一致）

### 3.2 Storage RLS 策略

> **关键**：Storage 的 RLS 策略和数据库 RLS 是**分开的**，需要分别配置。

检查路径：Supabase Dashboard → Storage → `resumes` bucket → Policies

**必须存在的策略：**

| 策略名 | 作用 | 预期 SQL |
|--------|------|---------|
| `resumes_select` | 读取（下载/预览） | `auth.role() = 'authenticated'` 或 `auth.uid() = user_id` |
| `resumes_insert` | 上传文件 | `auth.role() = 'authenticated'` |
| `resumes_delete` | 删除文件 | `auth.role() = 'authenticated'` 或 `auth.uid() = user_id` |

**如果策略缺失**，所有 Storage 操作都会被拒绝。

**常见错误**：用户可能只配了数据库 `profiles` 表的 RLS 策略，忘了配 Storage bucket 的 RLS 策略。

### 3.3 数据库 `profiles` 表 RLS

检查路径：Supabase Dashboard → Table Editor → `profiles` → Policies

- [ ] `profiles_select`：允许已认证用户读取自己的记录
- [ ] `profiles_update`：允许已认证用户更新自己的记录
- [ ] `profiles_insert`：允许注册时自动插入记录

### 3.4 Supabase Auth 配置

- [ ] **Site URL**：`https://woo-job.vercel.app`（Vercel 生产域名，**不是 localhost**）
- [ ] **Redirect URLs**：必须包含 `https://woo-job.vercel.app/auth/callback`
- [ ] 如果有自定义域名，也要加进去

### 3.5 Supabase Storage CORS（Allowed Origins）

检查路径：Supabase Dashboard → Storage → Settings → Allowed Origins

- [ ] **包含** `http://localhost:3000`（本地开发）
- [ ] **包含** `https://woo-job.vercel.app`（Vercel 生产）
- [ ] 如果有 Vercel preview 部署，也要加 `https://*.vercel.app`

## 四、Vercel 配置检查清单

### 4.1 环境变量（最常见根因）

路径：Vercel Dashboard → 项目 → Settings → Environment Variables

| 变量名 | 预期值 | 必须非 Sensitive |
|--------|--------|-----------------|
| `NEXT_PUBLIC_SUPABASE_URL` | `https://<project>.supabase.co` | ✅ |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `eyJhbG...` | ✅ |
| `SUPABASE_SERVICE_ROLE_KEY` | `eyJhbG...` | ❌（可保持 Sensitive） |

> ⚠️ **`NEXT_PUBLIC_` 前缀的变量绝对不能设为 Sensitive**，否则客户端代码无法读取，导致 Supabase URL 为 `undefined`，请求发到 Vercel 自身而非 Supabase。

**如果之前已创建但设为 Sensitive：**
1. 删除旧变量
2. 重新创建（**不勾选 Sensitive**）
3. 确保三个环境（Production / Preview / Development）都勾选
4. Redeploy

### 4.2 部署后验证

修改环境变量后必须 Redeploy：
- 方法 1：push 一个新 commit
- 方法 2：Deployments → 最新部署 → ⋯ → Redeploy

## 五、浏览器端调试步骤（按顺序执行）

### 步骤 1：确认环境变量已正确注入

在 Vercel 站点的浏览器控制台执行：

```javascript
console.log('SUPABASE_URL:', window.NEXT_PUBLIC_SUPABASE_URL);
console.log('SUPABASE_ANON_KEY:', window.NEXT_PUBLIC_SUPABASE_ANON_KEY ? '✅ configured' : '❌ missing');
```

**预期：** 两者都有值。如果 URL 是 `undefined` → Vercel 环境变量问题未解决。

### 步骤 2：确认 Supabase Session Cookie 存在

```javascript
document.cookie.split(';').map(c => c.trim()).filter(c => c.startsWith('sb-'))
```

**预期：** 至少有一个以 `sb-` 开头的 cookie（如 `sb-xxx-auth-token`）。

- 如果**没有** → Auth Callback 没有正确写入 cookie → 检查第五节
- 如果**有** → Cookie 存在，可能是值无效或已过期

### 步骤 3：检查 Supabase 客户端状态

在控制台执行（需要页面已加载 Supabase SDK）：

```javascript
// 检查当前 session
const sb = window.__SUPABASE_BROWSER_CLIENT__;
if (sb) {
  sb.auth.getSession().then(s => console.log('Session:', s));
} else {
  console.log('No Supabase client found');
}
```

### 步骤 4：确认登录状态

```javascript
window.__SUPABASE_BROWSER_CLIENT__?.auth.getUser().then(u => console.log('User:', u));
```

**预期：** 返回用户对象。如果报错或返回 null → session 无效。

## 六、Auth Callback 问题排查

如果步骤 2 没有看到 cookie，按以下顺序排查：

### 6.1 确认 OAuth 回调 URL 正确

登录时，观察地址栏跳转过程：

```
GitHub OAuth → 
https://github.com/login/oauth/authorize?...
  → Supabase OAuth → 
  → https://<project>.supabase.co/auth/v1/callback?code=xxx
    → 你的应用 → 
    → https://woo-job.vercel.app/auth/callback?code=xxx
```

在 Vercel 站点的 Network 面板找到 `/auth/callback` 请求：

1. 查看该请求的 **Response Headers** 中的 `Set-Cookie`
2. 确认包含 `sb-xxx-auth-token`，且有正确的 `Domain`、`Path`、`Expires`
3. 检查 `SameSite` 属性（应该是 `Lax` 或 `None`，**不应该是 `Strict`**）

### 6.2 检查 Supabase Site URL 配置

Supabase Dashboard → Authentication → URL Configuration：

- **Site URL**：必须设为 `https://woo-job.vercel.app`（不是 localhost）
- **Redirect URLs**：必须包含 `https://woo-job.vercel.app/auth/callback`

如果 Site URL 是 `localhost`，Supabase 重定向时会跳转到本地，Vercel 部署的版本永远收不到回调。

### 6.3 Cookie 属性验证

手动检查 Network 面板里 `/auth/callback` 响应的 `Set-Cookie`：

| 属性 | 预期值 |
|------|--------|
| `Domain` | `.woo-job.vercel.app` 或 `woo-job.vercel.app` |
| `Path` | `/` |
| `SameSite` | `Lax`（跨域场景建议） |
| `Secure` | `true`（生产环境必须有） |
| `HttpOnly` | `true`（安全最佳实践） |

## 八、紧急修复方案（如果排查时间过长）

如果暂时无法解决 session 问题，可以使用 **Service Role Key** 方案作为过渡（不推荐长期使用）：

1. 在 Server Action（`updateUserResume`）中用 Service Role Key 直接上传文件，绕过 RLS
2. 将 `updateUserResume` 改为使用 `createServerSupabaseClient` with `SUPABASE_SERVICE_ROLE_KEY`

> ⚠️ **警告**：Service Role Key 拥有数据库完整权限，不能暴露给客户端。仅在 Server Action 中使用，且 `SUPABASE_SERVICE_ROLE_KEY` 必须设为 **Sensitive** 环境变量。
