## 一、严重问题（高优先级）

### 1. `DraftPreviewCard` 缺失导致的运行时崩溃

**文件：** `components/AISidebar.tsx` (第 568 行)

**问题：** `AISidebar` 导入了 `DraftPreviewCard` 组件，但该文件位于错误路径。

```startLine:8:components/AISidebar.tsx
import DraftPreviewCard from './DraftPreviewCard';
```

然而 `DraftPreviewCard.tsx` 实际存放在 `components/DraftPreviewCard.tsx`，导入路径 `./DraftPreviewCard` 无法解析。这意味着当 AI 解析出结构化数据时，界面会直接崩溃，报 `Module not found` 错误。

**修复方式：** 将 `app/actions/ai.ts` 中返回的 `parsedData`（AIParsedData 类型）通过 props 传入 `DraftPreviewCard`，而非在 `AISidebar` 中直接渲染。或者确保导入路径正确。

**影响：** AI 副驾"建岗位"/"建任务"模式的确认卡片完全不可用，属于功能性阻断问题。

### 2. `updateJobStage` 回滚逻辑丢失

**文件：** `store/useJobStore.ts` (第 330-341 行)

**问题：** 拖拽卡片到新列时，`updateJobStage` 在失败时只回滚了 `jobs`，没有回滚 `trashedJobs`。

```startLine:330:store/useJobStore.ts
updateJobStage: async (id, newStage) => {
  const prev = get().jobs;
  set((s) => ({
    jobs: s.jobs.map((j) => (j.id === id ? { ...j, stage: newStage as JobStage } : j)),
  }));
  const result = await jobsActions.updateJobStage(id, newStage);
  if (result.error) {
    set({ jobs: prev }); // ← trashedJobs 未回滚
  }
},
```

对比 `trashJob` 的正确实现：

```startLine:435:store/useJobStore.ts
trashJob: async (id) => {
  const prev = get().jobs;
  set((s) => ({
    jobs: s.jobs.filter((j) => j.id !== id),
    trashedJobs: [{ ...job, stage: '已结束' as JobStage }, ...s.trashedJobs],
  }));
  const result = await jobsActions.trashJob(id);
  if (result.error) {
    set({ jobs: prev }); // ← trashedJobs 同样未回滚，逻辑一致但同样有问题
  }
},
```

**修复方式：** `updateJobStage` 应同时保存 `trashedJobs` 的快照并在失败时一并回滚。

### 3. `ProfileActionResult` 类型定义与实现不一致

**文件：** `types/database.ts` (第 210-212 行) vs `app/actions/profile.ts`

**问题：** `ProfileActionResult` 类型声明：

```startLine:210:types/database.ts
export type ProfileActionResult =
  | { profile: DbProfile }
  | { error: string };
```

但 `getProfile`/`updateProfile` 的返回类型实际是 `{ profile?: DbProfile; error?: string }`，两者的语义完全不同。`ProfileActionResult` 这个联合类型在代码库中没有任何地方使用，类型文件中的这个定义是无效的死代码。

**修复方式：** 删除 `types/database.ts` 第 210-212 行的 `ProfileActionResult` 类型定义（已无人引用）。

### 4. `deleteStorageFile` 从未被调用

**文件：** `app/actions/profile.ts` (第 266-279 行)

**问题：** 该 Server Action 定义了 `deleteStorageFile` 函数，但在整个代码库中没有任何地方调用它。Storage 文件删除逻辑在 `deleteUserResume` 中已有实现，`deleteStorageFile` 是重复代码。

**修复方式：** 删除该函数，或将其作为内部 helper 函数在 `deleteUserResume` 中复用。

**问题：** `createJob` 失败时返回中文错误消息，API 层返回英文错误消息，混合使用影响调试体验和用户理解。

## 二、中等问题（中优先级）

### 6. 看板列标题添加按钮视觉比重失衡

**修复方式：** 改为与简历添加按钮一致的幽灵风格：


**修复方式：** 删除 `isTaskInTimeRange` 函数，或确认其用途后统一使用。

### 8. `Next24HoursTask` 接口重复定义

**文件：** `lib/dateUtils.ts` (第 161-172 行)

**问题：** `Next24HoursTask` 接口与 store 中的 `Task` 接口几乎完全相同，只是字段名称略有不同（`date` vs `date`，`time` vs `time`）。在 `getNext24HoursTasks` 中使用独立接口导致类型转换复杂化。

**修复方式：** 直接使用 `Task` 类型，无需额外的接口包装。

### 9. `supabase-server.ts` 中 `createServerSupabaseClientSync` 永远返回空 cookie

**文件：** `lib/supabase-server.ts` (第 37-50 行)

```startLine:37:lib/supabase-server.ts
export function createServerSupabaseClientSync() {
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return []; // ← 永远返回空，永远没有 session
        },
        setAll() {},
      },
    }
  );
}
```

**问题：** 这个函数在所有 Route Handler 中都会创建出一个没有 session 的 Supabase 客户端，所有需要用户身份的操作都会静默失败。如果代码库中有使用此函数的地方，会导致权限绕过或数据访问错误。

**修复方式：** 如果有 Route Handler 使用了此函数，替换为异步版本；如果确认没有使用，删除该函数。

### 10. 颜色值硬编码导致维护成本高

**文件：** 全局

**问题：** 品牌主色 `#8B735B` 出现了超过 40 次，均为手动硬编码字符串。缺少统一的 Design Token 管理，一旦品牌色变更需要全局搜索替换。

**涉及文件：** `BottomShelf.tsx`、`KanbanColumn.tsx`、`TaskDetails.tsx`、`AgendaView.tsx`、`JobCard.tsx`、`AISidebar.tsx`、`page.tsx` 等几乎所有 UI 组件。

**修复方式：** 在 `tailwind.config.ts` 中定义 design tokens：

```ts
colors: {
  brand: {
    DEFAULT: '#8B735B',
    light: '#9D8B6F',
    dark: '#7A654D',
    bg: '#E5E1DA',
  },
}
```

然后将所有 `#8B735B` 替换为 `text-brand`、`bg-brand` 等语义化 token。

### 11. `AgendaView` 中"今天"任务排序逻辑错误

**文件：** `components/AgendaView.tsx`

### 14. `TrashDrawer` 重复拉取数据

**文件：** `app/page.tsx` (第 251-253 行)

```startLine:251:app/page.tsx
onClick={() => {
  fetchTrashedJobs(); // ← 每次打开都重新拉取
  setIsTrashOpen(true);
}}
```

每次打开回收站抽屉都会重新请求数据，即使数据刚刚已经加载过。建议在 store 中缓存上次加载时间，仅当数据过期（如超过 30 秒）才重新拉取。

### 15. `reorderJobs` 中 splice 操作副作用风险

**文件：** `store/useJobStore.ts` (第 367-377 行)

```startLine:367:store/useJobStore.ts
const [movedJob] = sourceColumn.splice(sourceIndex, 1);
// ...
sourceColumn.splice(destinationIndex, 0, movedJob);
```

`splice` 直接修改了 `sourceColumn` 数组（来自 `.filter()` 创建的新数组，无外部引用），逻辑上安全但代码可读性差。后续还有对 `sourceColumnUpdated` 的 map 操作，容易与原始数据混淆。

**修复方式：** 改用非 mutation 的 slice + spread 写法，提升可读性。

### 17. `signOutAction` 返回类型为 `void` 但实际可能抛出异常

**文件：** `app/actions/profile.ts` (第 281-286 行)

```startLine:281:app/actions/profile.ts
export async function signOutAction(): Promise<void> {
  const supabase = await createServerSupabaseClient();
  await supabase.auth.signOut();
  redirect('/login'); // ← 如果在 Client Component 中调用会抛出异常
}
```

`redirect()` 是 Next.js 的特殊异常机制，在 Client Component 中调用会导致渲染崩溃（已确认 `page.tsx` 第 321 行是在 `onClick` 回调中调用，这是允许的）。但类型声明为 `void` 掩盖了这个特殊行为，建议改为 `never` 返回类型以明确语义。

## 四、正面评价（值得保持）

1. **乐观更新模式设计良好：** Store 中大量使用了乐观更新 + 失败回滚机制，用户体验流畅，这是业界最佳实践。
2. **类型分层清晰：** `types/database.ts` 区分了 `DbJob`（数据库行）和 `JobWithTags`（API 层），`store/useJobStore.ts` 又有 UI 层的 `Job` 接口，三层类型各司其职。
3. **Zustand 订阅优化：** 大部分 store selector 使用了细粒度订阅（如 `(s) => s.jobs`），而非整个 store 订阅，减少了不必要的重渲染。
4. **DRY 原则应用较好：** `_helpers.ts` 集中了所有类型转换逻辑，避免了在多个 Server Action 中重复。
5. **错误处理覆盖率高：** 几乎所有 async 函数都有 try-catch 包装。
6. **Loading 状态管理完善：** Store 有 `isLoading`、`loadingMonth` 等状态，防止重复请求。
