## 一、用户调研方案（Discovery Interview Prep）

### 1.1 调研目标

**主要目标**：验证"AI 面试模拟"是否比现有方案更有效地帮用户通过面试

**核心问题**：
- 求职者面试准备的真实痛点是什么？
- 现有解决方案（牛客网、ChatGPT、真人 mock）的不足在哪里？
- 用户对"多 Agent 面试模拟"这个概念的真实反应是什么？

### 1.2 目标用户画像

| 用户类型 | 特征 | 优先级 |
|----------|------|--------|
| **互联网求职者（初级）** | 0-3 年经验，应届/跳槽，技术岗/产品岗 | **P0 — 优先访谈** |
| **互联网求职者（中级）** | 3-5 年经验，跳槽为主，有面试经验 | P1 |
| **转行求职者** | 非互联网转互联网，面试准备焦虑最高 | P1 |
| **面试官/HR** | 从招聘方角度理解"什么是好回答" | P2 — 补充视角 |

### 1.3 调研约束

- **样本量**：8-12 人（5 人即可看到模式）
- **时间**：1-2 周完成
- **招募渠道**：脉脉、牛客网讨论区、小红书求职帖、微信群、朋友圈
- **激励**：每人 30-50 元红包或咖啡券

### 1.4 访谈方法

**采用 Problem Validation + JTBD 混合方法**

理由：
- Problem Validation（Mom Test 风格）：验证痛点是否真实
- JTBD：理解用户"雇佣"面试练习工具的深层动机

---

### 1.5 访谈问题设计

#### 开场（5 分钟）

```
"感谢你抽时间。我在研究互联网求职者如何准备面试，这不是推销，
我只是想了解你的真实经历。没有标准答案，越坦诚越有帮助。"
```

#### 核心问题（30-40 分钟）

**第一组：痛点验证（Problem Validation）**

| # | 问题 | 追问 | 避免 |
|---|------|------|------|
| 1 | "能讲讲你最近一次准备面试的经历吗？从什么时候开始准备的？" | "准备了多长时间？最后结果如何？" | 不要问"你觉得面试准备难吗？"（引导性） |
| 2 | "准备过程中，哪个环节让你最头疼？" | "具体是什么让你觉得难？能举个例子吗？" | 不要问"你是不是觉得刷题很烦？" |
| 3 | "你用了哪些方法或工具来准备？" | "每个工具你用了多久？为什么放弃/继续用？" | 不要问"你用过 XX 吗？"（主动提及竞品） |
| 4 | "有没有哪个问题是你准备了但面试时还是答不好的？" | "当时什么感觉？后来怎么处理的？" | 不要问"你是不是不知道怎么回答行为面试题？" |
| 5 | "如果你能回到准备面试的那天，你会做什么不同的事？" | "为什么？那件事对你的面试结果影响大吗？" | 不要问"你需要一个 AI 面试工具吗？" |

**第二组：JTBD 深挖（理解动机）**

| # | 问题 | 追问 | 避免 |
|---|------|------|------|
| 6 | "你当时为什么决定要准备这次面试？是什么驱动你的？" | "是焦虑？还是有明确的目标？" | 不要问"你是不是很想拿到 offer？" |
| 7 | "你理想中的面试准备过程是什么样的？" | "和你实际经历的差距在哪里？" | 不要问"你想要一个 AI 模拟面试吗？" |
| 8 | "你有没有找人 mock 过？效果怎么样？" | "如果没找过，为什么？如果找过，和自己练有什么不同？" | 不要问"你觉得 AI 模拟会比真人好吗？" |
| 9 | "面试结束后，你是怎么判断自己表现好不好的？" | "你从哪里得到反馈？反馈够用吗？" | 不要问"你需要实时评分吗？" |
| 10 | "你身边有没有面试准备做得特别好的人？他们是怎么做的？" | "你能学到什么？为什么你没那样做？" | — |

**第三组：概念测试（最后 5 分钟，可选）**

> 注意：只在前 25 分钟结束后，简短提及概念，不要推销。

```
"我有个想法想听听你的反应：如果有一个 AI 工具，可以模拟不同
风格的面试官（比如技术面、HR 面），根据你的简历生成个性化问题，
回答后给你具体的改进建议——你会有兴趣试试吗？"
```

**观察重点**：
- 用户的第一反应是"太好了"还是"嗯...还行"？
- 用户主动追问的功能是什么？（那才是真需求）
- 用户的顾虑是什么？（价格？准确度？隐私？）

#### 收尾（5 分钟）

```
"总结一下我听到的：[关键洞察]。我理解得对吗？
另外，你身边有没有也在准备面试的朋友？可以介绍给我聊聊吗？"
```

### 1.6 访谈执行清单

| 项目 | 详情 |
|------|------|
| **每场时长** | 45 分钟 |
| **记录方式** | 征得同意后录音 + 关键词笔记 |
| **每日上限** | 2-3 场（需要时间消化） |
| **招募话术** | "我在做面试准备相关的研究，想听听你的真实经历，30 分钟聊完送 XX 红包" |
| **招募目标** | 联系 20-30 人 → 约到 8-12 人 → 完成 8 场以上 |

### 1.7 访谈成功标准

- 听到**具体故事**，而非泛泛抱怨："上周三我花了 3 小时..."
- 发现**过去行为**，而非假设愿望："我试过 XX，但放弃了因为..."
- 3+ 场访谈出现**相同模式**
- 至少有 **1 个意外发现**（如果全在预期内，说明问了引导性问题）
- 能**引用用户原话**

### 1.8 数据合成方法

每场访谈后立即记录：
1. 用户的核心痛点（用原话）
2. 用户的现有解决方案
3. 用户未被满足的需求
4. 用户对概念的反应（如果测试了）

5 场后开始看模式，用便利贴/表格聚类。

## 二、机会-解决方案树（Opportunity Solution Tree）

### 2.1 期望成果（Desired Outcome）

```
帮助互联网求职者将面试通过率提升
```

**可衡量目标**：用户使用 WooJob 后，自评面试准备充分度从 4/10 提升到 7/10

### 2.2 机会地图

基于假设（待调研验证），以下是可能的机会点：

```
期望成果：提升面试通过率
    │
    ├── 机会1：用户不了解目标岗位的真实考察重点
    │     │
    │     ├── 方案A：基于岗位 JD 自动生成考察重点清单 ← 轻量，可快速验证
    │     ├── 方案B：RAG 题库 + 岗位匹配推荐问题
    │     └── 方案C：多 Agent 模拟不同面试官视角 ← 多 Agent 在此有价值
    │
    ├── 机会2：用户回答缺乏结构化框架，不知道"好回答"长什么样
    │     │
    │     ├── 方案A：STAR/SCQA 答题框架提示 ← 最轻量
    │     ├── 方案B：标准答案要点对比（你的方案）
    │     └── 方案C：AI 实时分析回答并给出结构化建议
    │
    └── 机会3：用户无法获得有效反馈，不知道自己"差在哪"
          │
          ├── 方案A：回答后 AI 给出定性反馈（优点+改进点）← MVP 推荐
          ├── 方案B：多维度评分系统（你的方案）
          └── 方案C：模拟面试官追问，暴露薄弱环节 ← 多 Agent 在此有价值
```

### 2.3 各机会点深度分析

#### 机会1：不了解岗位考察重点

**用户痛点假设**：
- "我不知道产品经理面试到底考什么"
- "我准备了很多，但面试问的完全不是我准备的"

**验证问题**：
- 你上次面试前，是怎么了解岗位要求的？
- 你准备的内容和实际面试问的差距大吗？

**解决方案评估**：

| 方案 | 可行性 | 影响力 | 市场匹配 | 多Agent价值 |
|------|--------|--------|----------|-------------|
| A. 岗位考察清单 | 5 | 3 | 4 | 低 — 不需要多Agent |
| B. RAG 题库匹配 | 4 | 4 | 4 | 低 — 单Agent即可 |
| C. 多Agent多视角 | 3 | 4 | 3 | **高 — 不同Agent模拟不同面试官关注点** |

---

#### 机会2：回答缺乏结构化框架

**用户痛点假设**：
- "我知道答案但说得乱七八糟"
- "我不知道什么样的回答算好"

**验证问题**：
- 你回答面试题时，有没有固定的思考框架？
- 你觉得自己回答的最大问题是什么？

**解决方案评估**：

| 方案 | 可行性 | 影响力 | 市场匹配 | 多Agent价值 |
|------|--------|--------|----------|-------------|
| A. 答题框架提示 | 5 | 3 | 4 | 低 |
| B. 要点对比 | 4 | 4 | 4 | 低 |
| C. AI实时分析 | 3 | 4 | 4 | 中 — 可以用专门的分析Agent |

---

#### 机会3：无法获得有效反馈

**用户痛点假设**：
- "我练了很多但不知道自己到底差在哪"
- "找人 mock 太麻烦，而且对方也不专业"

**验证问题**：
- 你练习面试后，怎么判断自己答得好不好？
- 你有没有从别人那里得到过面试反馈？质量如何？

**解决方案评估**：

| 方案 | 可行性 | 影响力 | 市场匹配 | 多Agent价值 |
|------|--------|--------|----------|-------------|
| A. 定性反馈 | 5 | 4 | 5 | 低 |
| B. 多维评分 | 3 | 3 | 3 | 低 |
| C. 追问暴露薄弱 | 3 | 5 | 5 | **高 — 多Agent可以模拟不同追问风格** |

---

### 2.4 POC 推荐

#### 推荐方案：机会3 + 方案C（追问暴露薄弱环节）

**假设**：
> "如果 AI 面试官能根据用户回答进行有针对性的追问，暴露用户思考的薄弱环节，那么用户的面试准备效果将显著提升。"

**为什么选这个**：
1. **用户价值最高**：追问是真实面试中最难的部分，也是现有工具（题库、ChatGPT）做得最差的
2. **多 Agent 天然适配**：不同面试官有不同追问风格，多 Agent 架构在这里有真实的产品价值
3. **技术学习价值**：多 Agent 状态管理、上下文传递、追问逻辑，是很好的技术实践
4. **差异化最强**：牛客网是题库，ChatGPT 是自由对话，追问式模拟是一个未被满足的点

**实验方案**：
- **类型**：原型测试（手动模拟）
- **做法**：用 ChatGPT 手动扮演面试官，对 5 个用户进行追问式面试
- **观察**：用户是否觉得追问有帮助？追问质量如何？
- **成功标准**：4/5 用户反馈"比自己练习效果好"

## 三、多 Agent 架构的价值重新评估

你提到想学习多 Agent 技术并丰富简历。这是一个完全合理的目标，但需要让它**与产品价值对齐**，而非为了技术而技术。

### 3.1 多 Agent 在本产品中的真实价值点

| 价值点 | 说明 | 产品价值 | 技术学习价值 |
|--------|------|----------|-------------|
| **多角色追问** | 不同 Agent 模拟不同面试官的追问风格（技术深挖 vs 行为考察 vs 压力测试） | **高** — 这是用户真正需要的 | **高** — Agent 人格化、上下文管理 |
| **回答质量评估** | 专门的 Assessor Agent 分析回答质量 | **中** — 可以用单 Agent 实现 | **中** — 评分逻辑可以很复杂 |
| **问题生成策略** | Director Agent 根据用户水平动态调整问题难度和方向 | **高** — 个性化体验 | **高** — 条件路由、状态机 |
| **标准答案生成** | 独立 Agent 生成参考答案 | **低** — 单 Agent 即可 | **低** |
| **Cross-Fire 多人围攻** | 多个 Agent 同时提问 | **低** — 可能增加焦虑 | **中** — 并发 Agent 协调 |

### 3.2 建议：保留多 Agent，但精简到有产品价值的部分

```
精简后的多 Agent 架构（3 个 Agent）：

┌─────────────────────────────────────────────────┐
│                Director Agent                    │
│  职责：                                          │
│  - 解析岗位要求，确定考察重点                      │
│  - 根据用户回答质量调整问题难度                    │
│  - 决定何时切换面试官角色                          │
│  - 管理整体面试流程和状态                          │
└───────────┬─────────────────────┬───────────────┘
            │                     │
            ▼                     ▼
┌────────────────────┐  ┌────────────────────────┐
│  Interviewer Agent │  │   Assessor Agent       │
│  职责：             │  │   职责：                │
│  - 提问            │  │   - 分析回答质量        │
│  - 追问（核心价值） │  │   - 生成改进建议        │
│  - 模拟不同风格     │  │   - 提供结构化反馈      │
│  - 压力测试（可选） │  │   - 与要点对比          │
└────────────────────┘  └────────────────────────┘
```

**为什么这样设计**：
- **Director**：学习 LangGraph 状态机、条件路由、Agent 编排
- **Interviewer**：学习 Agent 人格化（system prompt 设计）、上下文管理、追问逻辑
- **Assessor**：学习 Agent 输出结构化评估、多维度打分

这三个 Agent 足以展示你的多 Agent 能力，同时每个都有明确的产品价值。

## 四、调整后的 MVP 建议

基于以上分析，建议 MVP 聚焦：

### Phase 1：核心验证（2 周）

```
功能：
1. 用户输入岗位 → 生成 5 道针对性面试题
2. 用户逐题回答 → AI 追问 1-2 轮（核心差异化）
3. 每题回答后 → 给出定性反馈（优点 + 改进点）
4. 结束后 → 生成简要报告

技术：
- Next.js 前端
- LangGraph 编排 Director + Interviewer + Assessor
- 简单 RAG（本地题库，不需要向量数据库）
- OpenAI/Claude API
```

### Phase 2：增强体验（2 周）

```
功能：
1. 简历上传 → 个性化问题生成
2. 追问深度提升（基于回答中的薄弱点）
3. 答题框架提示（STAR 等）
4. 面试报告增强（分维度反馈）

技术：
- RAG 升级（向量数据库 + 面经库）
- 简历解析模块
- 评分逻辑完善
```

### Phase 3：打磨上线（2 周）

```
功能：
1. 多岗位支持
2. 面试历史记录
3. UI/UX 打磨
4. 部署上线
```

**总计 6 周**，比原方案 8 周更紧凑，且每个 Phase 都有可验证的目标。
