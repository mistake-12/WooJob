## 一、产品概述

### 1.1 背景与机会

互联网求职者在求职过程中面临以下核心问题：

1. **能力与岗位不匹配**：想投 AI 产品经理但缺乏 AI 项目经历，不知道该补什么
2. **准备碎片化**：简历优化、面试练习、投递策略各自为战，缺乏系统性
3. **反馈缺失**：练了很多面试题，但不知道自己到底差在哪
4. **全程无陪伴**：从投简历到拿 offer 的漫长过程中，没有人指导

现有竞品（牛客网、力扣、ChatGPT）只覆盖了"面试练习"这一个环节，且缺乏个性化和持续陪伴。

### 1.2 产品定义

```
For 正在求职的互联网从业者（0-5 年经验）
that need 系统性的求职指导和持续陪伴
WooJob is a AI 求职陪跑教练
that 从能力诊断到面试模拟到 Offer 谈判，提供全程个性化指导
Unlike 牛客网只提供题库、猎聘只提供职位信息
WooJob provides 像一个资深猎头一样陪你走完求职全程
```

### 1.3 目标用户

| 用户画像 | 特征 | 核心需求 | 优先级 |
|----------|------|----------|--------|
| **应届求职者** | 0-1 年经验，校招，焦虑感强 | 系统性指导，不知道从哪开始 | P0 |
| **跳槽者（初级）** | 1-3 年经验，想换公司/方向 | 能力诊断 + 面试准备 | P0 |
| **转行者** | 非互联网转互联网，或跨岗位 | 差距填补 + 学习路径规划 | P0 |
| **跳槽者（中级）** | 3-5 年经验，有明确目标 | 面试精进 + Offer 谈判 | P1 |

### 1.4 成功指标

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 用户自评准备充分度 | 使用前 4/10 → 使用后 7/10 | 用户调研 |
| 面试通过率 | 提升 30%（vs 用户历史） | 用户自报 |
| 7 日留存率 | > 40% | 埋点数据 |
| 完整旅程完成率 | > 25% | 埋点数据 |
| NPS | > 40 | 用户调研 |

## 二、产品架构

### 2.1 架构理念：Hub & Spoke（中心辐射）

采用**中心辐射**架构取代线性流水线。用户从中心 Hub 自由选择进入任何阶段，各阶段可独立运行，有上游数据时自动增强体验。

```
        ┌────────────┐
        │  能力诊断   │
        │  🔍 2-5min │
        └─────┬──────┘
              │ 输出: 差距清单
              ▼
        ┌────────────┐         ┌────────────┐
        │  差距填补   │         │  简历优化   │
        │  📚 1-4周  │────────→│  📄 15-30m │
        └────────────┘ 新经历   └─────┬──────┘
              │                       │
              │    ┌──────────┐       │
              └───→│   Hub    │◄──────┘
              ┌───→│  中心    │
              │    └──────────┘
              │         │
        ┌────────────┐  │  ┌────────────┐
        │  面试模拟   │  │  │  投递策略   │
        │  🎤 持续   │◄─┘  │  🎯 15-30m │
        └────────────┘     └─────┬──────┘
                                 │
                           ┌────────────┐
                           │  Offer谈判  │
                           │  💰 15-30m │
                           └────────────┘
```

### 2.3 阶段依赖关系

各阶段之间为**弱依赖**关系：有上游数据时进入增强模式，没有时降级为通用模式。

```
┌─────────┐     依赖关系图
│ 能力诊断 │ ─────────────────────────────────────┐
└─────────┘                                      │
     │ 输出: 差距清单                              │
     ▼                                            │
┌─────────┐                                      │
│ 差距填补 │ ◄── 硬依赖：能力诊断（需要差距清单）    │
└─────────┘                                      │
     │ 输出: 新增经历                              │
     ▼                                            │
┌─────────┐                                      │
│ 简历优化 │ ◄── 软依赖：差距填补（新经历）          │
└─────────┘ ◄── 软依赖：能力诊断（差距方向）        │
     │                                            │
     ▼                                            │
┌─────────┐                                      │
│ 面试模拟 │ ◄── 软依赖：简历优化（简历内容）        │
└─────────┘ ◄── 软依赖：能力诊断（薄弱点）          │
     │                                            │
     ▼                                            │
┌─────────┐                                      │
│ 投递策略 │ ◄── 软依赖：能力诊断（匹配度）          │
└─────────┘                                      │
     ▼                                            │
┌─────────┐                                      │
│Offer谈判 │ ◄── 无依赖（独立输入）                 │
└─────────┘                                      │
```

**依赖规则表：**

| 阶段 | 硬依赖（必须有） | 软依赖（有更好） | 无依赖时的降级策略 |
|------|----------------|-----------------|------------------|
| 能力诊断 | 无 | 无 | — |
| 差距填补 | 能力诊断 | 无 | 引导用户先做诊断，或直接跳过 |
| 简历优化 | 无 | 能力诊断 + 差距填补 | 用户手动粘贴简历，AI 通用优化 |
| 面试模拟 | 无 | 简历 + 目标岗位 | 用户手动输入岗位，AI 通用提问 |
| 投递策略 | 无 | 能力诊断 | 用户手动输入背景，AI 通用建议 |
| Offer 谈判 | 无 | 无 | 用户手动输入 Offer 详情 |

**为什么不需要 Director？**

| 场景 | Director 方案 | Hub + Guide Agent 方案 |
|------|-------------|----------------------|
| 用户想进入面试模拟 | 用户说"我想练面试" → Director 识别意图 → 路由 | 用户点击"🎤面试模拟"卡片 → 直接调用 API |
| 检查依赖 | Director 检查硬依赖 | 前端在点击时检查，锁定状态已在 Hub 展示 |
| 用户想问"我该做什么" | Director 回答 | Guide Agent 回答（侧边栏常驻） |
| 额外 LLM 调用 | 每次路由消耗一次 Director 调用 | 无额外调用（点击 = 路由） |

**Guide Agent 的核心价值：**

```
Guide Agent 不是路由层，而是陪伴层：

用户在 Hub 页面：
  → Guide Agent："👋 你已完成诊断，建议试试面试模拟"
  → 渐进式披露，引导用户使用产品

用户在面试模拟中：
  → 用户侧边栏问："我的薄弱项有哪些？"
  → Guide Agent 调用 readGapAnalysis()，回答用户
  → 不打断面试流程，随时可问

用户回到 Hub：
  → Guide Agent："✅ 面试完成！推荐下一步：📄简历优化"
  → 基于进度推荐，而非强制路由
```

**子 Agent 通信机制：**

所有 Agent（包括 Guide Agent）平级，不直接通信，通过共享数据池中转：

```
面试 Agent 调用 readResume()
      │
      ▼
┌──────────────┐
│  共享数据池    │ ◄── 诊断 Agent 之前写入的简历数据
│  resume: {...}│
└──────────────┘
      │
      ▼
面试 Agent 获得原始简历 JSON（无压缩、无格式转换）

Guide Agent 调用 readJourneyProgress()
      │
      ▼
┌──────────────┐
│  共享数据池    │ ◄── 各 Agent 写入的阶段状态
│  stages: {...}│
└──────────────┘
      │
      ▼
Guide Agent 获得进度数据，推荐下一步
```

### 2.5 记忆架构

采用三层记忆模型，子 Agent 通过工具调用按需访问各层记忆：

```
┌─────────────────────────────────────────────────────────┐
│                    三层记忆架构                            │
│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  长期记忆（Long-term Memory）                     │    │
│  │  存储：PostgreSQL                                │    │
│  │  内容：用户画像、历史诊断、面试记录、学习进度        │    │
│  │  生命周期：永久，跨 session                        │    │
│  │  写入时机：阶段完成时同步                          │    │
│  │  读取时机：用户登录时加载到工作记忆                  │    │
│  └─────────────────────────────────────────────────┘    │
│                         ▲ 写入                          │
│                         │ 读取                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  工作记忆（Working Memory）= 共享数据池            │    │
│  │  存储：内存 / Zustand                             │    │
│  │  内容：当前旅程的 shared context                   │    │
│  │  生命周期：单次旅程（可能跨多个 session）            │    │
│  │  访问方式：子 Agent 通过工具调用按需读取             │    │
│  └─────────────────────────────────────────────────┘    │
│                         ▲ 写入                          │
│                         │ 读取                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  短期记忆（Short-term Memory）= 对话历史           │    │
│  │  存储：消息数组（内存）                            │    │
│  │  内容：当前阶段的对话消息                          │    │
│  │  生命周期：单个阶段，切换阶段后清空                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**记忆的读写时机：**

| 事件 | 短期记忆（对话历史） | 工作记忆（共享数据池） | 长期记忆（数据库） |
|------|-------------------|---------------------|------------------|
| 用户登录 | — | 从数据库加载历史数据到共享池 | 读取 |
| 点击某个阶段 | 清空（新阶段新对话） | 子 Agent 按需调用工具读取 | — |
| 对话进行中 | 每轮追加消息 | — | — |
| 阶段产出完成 | — | 产出物写入共享池 | — |
| 回到 Hub 页面 | — | 共享池保持不变 | 同步：共享池→数据库 |
| 用户退出 | 丢弃 | 丢弃（已同步到数据库） | 保存最终状态 |

**子 Agent 的工具调用示例：**

```typescript
// Interviewer Agent 可调用的工具
const interviewerTools = [
  {
    name: "read_resume",
    description: "获取候选人的完整简历数据",
    execute: async () => await db.getResume(userId)
  },
  {
    name: "read_gap_analysis",
    description: "获取能力诊断结果，包含各项能力的差距",
    execute: async () => await db.getGapAnalysis(journeyId)
  },
  {
    name: "read_high_priority_gaps",
    description: "只获取高优先级的能力差距，用于重点追问",
    execute: async () => {
      const analysis = await db.getGapAnalysis(journeyId);
      return analysis.gaps.filter(g => g.priority === 'high');
    }
  }
];
```

### 2.6 数据模型：共享数据池

各阶段产出物统一存入共享数据池，子 Agent 通过工具调用按需读取：

```typescript
interface JourneyContext {
  // 共享数据池——子 Agent 通过工具调用访问
  shared: {
    targetPosition?: string;       // 目标岗位
    targetCompany?: string;        // 目标公司
    resume?: ResumeData;           // 简历数据
    gapAnalysis?: GapAnalysis;     // 能力诊断结果
    newExperiences?: Experience[]; // 差距填补产出的新经历
  };

  // 各阶段独立产出
  stages: {
    diagnosis?: StageOutput;       // 诊断报告
    gapFilling?: StageOutput;      // 学习计划 + 进度
    resume?: StageOutput;          // 优化建议
    interview?: StageOutput[];     // 多次面试记录
    strategy?: StageOutput;        // 投递策略
    offer?: StageOutput;           // Offer 评估
  };

  // 进度追踪
  completedStages: StageType[];
  currentStage?: StageType;
}
```

### 2.7 数据流

**阶段内数据流（用户在某个阶段对话时）：**

```
用户输入
  │
  ▼
前端（Hub 页面路由，检查依赖）
  │
  ├── 直接调用子 Agent API（无需 Director 中转）
  │
  ▼
子 Agent（自治运行）
  │
  ├── 按需调用工具获取数据（readResume / readDiag / ...）
  │         │
  │         ▼
  │   共享数据池 / 数据库（返回原始数据，无损）
  │
  ├── LLM 生成回复（基于工具返回的原始数据 + 对话历史）
  │
  ├── 产出物写回共享数据池
  │
  └── 前端更新阶段状态 + 通知 Guide Agent 更新推荐
```

**Guide Agent 数据流（用户在侧边栏提问时）：**

```
用户在侧边栏提问（如"我的薄弱项有哪些？"）
  │
  ▼
Guide Agent
  │
  ├── 调用工具获取数据（readGapAnalysis / readJourneyProgress / ...）
  │         │
  │         ▼
  │   共享数据池 / 数据库
  │
  ├── LLM 生成回答（基于工具返回的数据）
  │
```

### 5.2 Agent 编排架构

采用 **无 Director + Guide Agent 侧边栏** 模式：

- **无 Director**：Hub 页面（前端）负责路由，用户点击卡片直接调用子 Agent API
- **子 Agent**：自带工具调用能力，按需从共享数据池/数据库拉取原始数据
- **Guide Agent**：常驻侧边栏，随身教练，负责数据查询、进度推荐、渐进式披露
- **通信方式**：所有 Agent 平级，不直接通信，通过共享数据池中转

```typescript
// API Route 示例（以面试模拟为例）
// app/api/agents/interviewer/route.ts
export async function POST(req: Request) {
  const { messages, context } = await req.json();
  
  // 子 Agent 自主决定是否调用工具
  const response = await callAgent({
    agent: 'interviewer',
    tools: [
      { name: 'read_resume', execute: () => db.getResume(userId) },
      { name: 'read_gap_analysis', execute: () => db.getGapAnalysis(journeyId) },
      { name: 'read_high_priority_gaps', execute: () => db.getHighPriorityGaps(journeyId) },
    ],
    systemPrompt: INTERVIEWER_PROMPT,
    messages,
  });
  
  return streamResponse(response);
}

// Guide Agent API Route
// app/api/agents/guide/route.ts
export async function POST(req: Request) {
  const { message } = await req.json();
  
  const response = await callAgent({
    agent: 'guide',
    tools: [
      { name: 'read_journey_progress', execute: () => db.getJourneyProgress(userId) },
      { name: 'read_gap_analysis', execute: () => db.getGapAnalysis(journeyId) },
      { name: 'recommend_next_stage', execute: () => recommendNext(userId) },
      { name: 'get_stage_intro', execute: (stage) => STAGE_INTRO[stage] },
    ],
    systemPrompt: GUIDE_PROMPT,
    messages: [...sidebarHistory, { role: 'user', content: message }],
  });
  
  return streamResponse(response);
}
```

### 5.3 记忆架构

三层记忆模型：

```
长期记忆（PostgreSQL）  ←→  工作记忆（Zustand/内存）  ←→  短期记忆（消息数组）
  用户画像、历史诊断         当前旅程共享数据池             当前阶段对话历史
  面试记录、学习进度         子 Agent 通过工具访问          切换阶段后清空
```

| 层级 | MVP 实现 | 正式版实现 |
|------|---------|-----------|
| 长期记忆 | localStorage（JSON） | PostgreSQL + Prisma |
| 工作记忆 | Zustand state | Zustand + 服务端状态同步 |
| 短期记忆 | 消息数组（内存） | 消息数组 + 滑动窗口摘要 |

### 6.3 关键交互流程

**自由链路流程（用户可从任何阶段进入）：**

```
场景A：用户从"面试模拟"直接进入（最常见的入口）

1. 用户在 Hub 页面点击"🎤 面试模拟"
2. 系统检查依赖：无硬依赖，有软依赖（简历/诊断）
3. 进入面试对话页面，AI 询问："你想面试什么岗位？有简历吗？"
4. 用户回复："AI 产品经理，这是我简历：[粘贴]"
5. 子 Agent 调用工具解析简历，开始模拟面试
6. 5 轮问答 + 追问，侧边栏实时显示评分
7. 面试结束，生成面试报告（InterviewReportCard）
8. 用户点击"返回 Hub"，回到选择中心
9. Hub 页面更新：面试模拟状态变为 ✅


场景B：用户从"能力诊断"进入（完整旅程起点）

1. 用户在 Hub 页面点击"🔍 能力诊断"
2. 系统询问目标岗位，用户上传简历
3. 子 Agent 调用工具解析 JD 和简历，生成诊断报告
4. 诊断结果写入共享数据池（工作记忆）
5. 系统提示："诊断完成！你可以继续其他阶段，或返回 Hub"
6. 用户返回 Hub，看到诊断卡片变为 ✅
7. 差距填补卡片从 🔒 变为 ○（解锁）


场景C：用户想切换阶段

1. 用户在面试模拟中，想看看自己的诊断结果
2. 点击"返回 Hub"（或侧边栏快捷入口）
3. 回到 Hub 页面，点击"🔍 能力诊断"
4. 诊断 Agent 调用工具读取历史诊断数据
5. 展示之前的诊断报告，用户可重新诊断
```

```typescript
  // locked: 硬依赖未满足

  // active: 进行中
  // completed: 已完成
  startedAt?: Date;
  completedAt?: Date;

}

  | 'gap_filling' 
  | 'resume' 
  | 'interview' 
  | 'strategy' 
  | 'offer';
```

## 九、风险与应对

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| LLM 回答质量不稳定 | 高 | 中 | 多轮 prompt 优化 + 人工审核样本 |
| 用户中途流失 | 高 | 高 | 阶段产出物即时可见 + 进度激励 |
| 评分准确性不足 | 中 | 中 | 初期定性反馈为主，逐步引入定量评分 |
| 学习计划过于泛化 | 中 | 中 | 基于用户具体差距生成，非通用模板 |
| 面试题库冷启动 | 中 | 低 | 先用 LLM 生成 + 公开数据集 |
| 成本过高 | 中 | 中 | 缓存上下文 + 选择性使用 GPT-4o |
