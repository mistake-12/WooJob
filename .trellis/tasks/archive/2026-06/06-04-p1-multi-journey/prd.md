# P1: 多陪跑记录管理 - 切换/新建/改名

## Goal

在求职陪跑主画布的右上角添加"陪跑记录"管理功能，让用户能创建、切换、重命名多条求职陪跑旅程，而不是像 MVP 那样只能有一条。

## What I already know

* MVP 阶段 `getOrCreateJourney()` 每个用户只维护一条 journey
* 当前 Hub header 是静态的标题 "选择你需要的陪跑阶段"
* `ai_journeys` 表已有 `title` 字段（当前固定值为 '求职陪跑'）
* Hub 目前只有标题文案，右上角没有交互元素
* P0 MVP 讨论时已把"按目标岗位/目标方向创建多个 journey"列为 P1 迭代方向

## Requirements (evolving)

* 在 Hub 页面（无阶段选中时）的右上角区域添加陪跑记录选择器
* 选择器展示当前 journey 名称，点击后下拉显示全部 journey 列表
* 每条记录可点击切换；当前选中的记录高亮
* 支持新建陪跑：输入名称后创建新 journey
* 支持改名：内联编辑或弹窗修改当前选中的 journey 标题
* 选择不同 journey 后，卡片状态、诊断报告、行动计划全部切换到对应 journey 的数据
* 保持原有单页应用架构不变

## Acceptance Criteria (evolving)

* [ ] Hub 右上角有陪跑记录选择器（dropdown 风格）
* [ ] 可以显示所有陪跑记录列表
* [ ] 可以创建新陪跑记录
* [ ] 可以重命名陪跑记录
* [ ] 切换记录后卡片状态正确刷新

## Definition of Done

* TypeScript 编译通过
* Build 通过
* 不影响看板/日程视图

## Out of Scope (explicit)

* 删除陪跑记录
* 陪跑记录分享/导出
* 跨用户协作

## Decisions

* **交互形态**：采用 Dropdown 下拉菜单。右上角显示当前陪跑名称 + 箭头，点击展开列表；列表项显示名称 + 选中态高亮；底部有"新建陪跑"入口；每条记录支持内联改名。

* **新建策略**：纯新建空白陪跑，只需输入名称即可创建空 journey，从零开始做诊断和计划。不复用、不自动继承现有数据。

## Open Questions

* **改名策略**：点击当前陪跑名称直接触发内联编辑，回车确认；下拉列表中的其他记录不显示改名入口，改名非当前陪跑需先切换。

## Open Questions

* **布局位置**：选择器与标题区同一行，右对齐。左侧标题+描述，右侧陪跑选择器。不额外占用纵向空间。
