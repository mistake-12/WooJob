## 一、 产品愿景与痛点分析

### 1.1 业务背景
在秋招/春招的高压环境下，求职者平均需要投递 50-100 家企业。传统的 Excel 或备忘录无法解决进度混乱、时间线冲突、以及面试准备无从下手的问题。

### 1.2 核心痛点与解决方案
*   **痛点 1：状态流转与死线焦虑。** ➡️ **方案：** 双视图引擎（看板掌控宏观进度，日程管理微观 Deadlines）。
*   **痛点 2：信息录入繁琐，反复填表。** ➡️ **方案：** AI 工作流 1（根据照片截图如邮箱、岗位信息，一键解析录入）。
*   **痛点 3：岗位搜寻耗时，信息茧房。** ➡️ **方案：** AI 工作流 2（异步岗位爬取与 RAG 智能推荐）。
*   **痛点 4：面试无侧重点，缺乏实战。** ➡️ **方案：** AI 工作流 3（基于简历+JD的双向深度模拟面试）。

## 二、 系统整体架构设计 (全栈)

产品采用前后端彻底分离的微服务架构思想。

*   **前端展示层 (Client):** Next.js 14 + React + Tailwind CSS。采用状态提升 (Lifting State Up) 和乐观更新 (Optimistic UI) 保障极致的拖拽与交互丝滑度。
*   **后端服务层 (Server API):** Node.js / Serverless API Routes。提供标准 RESTful 接口。
*   **数据持久层 (Database):** PostgreSQL (Supabase)。包含 `Users`、`Jobs`、`Tasks` 核心业务表。
*   **AI 驱动层 (Agent Engine):** 接入 DeepSeek/OpenAI 大模型，配合 Vector DB (向量数据库) 实现语义检索与多轮对话。

## 三、 核心业务功能需求 (基础前后端)

### 3.1 核心视图 1：敏捷求职看板 (Kanban View)
*   **前端交互：** 
    *   基于 `@hello-pangea/dnd` 实现 6 大阶段列的自由拖拽。
    *   **物理动效：** 拖拽时卡片具有 `rotate-2 scale-105` 的吸附反馈。
    *   **动态进度条：** 卡片底部进度条不仅跟随“大阶段”，在处于“面试中”时，根据具体的“初面/技术面/HR面”进行细腻的百分比递增。
*   **后端支撑：** 
    *   `GET /api/v1/jobs`：拉取用户看板数据。
    *   `PATCH /api/v1/jobs/{id}/stage`：拖拽松手瞬间触发，局部更新卡片状态。

### 3.2 核心视图 2：待办日程引擎 (Agenda View)
*   **前端交互：** 
    *   左侧迷你日历与右侧时间轴列表联动。采用**“前端一次性拉取 + 本地实时过滤”**方案，点击日历实现 0 延迟的任务筛选。
    *   单卡片交互：点击左侧圆圈完成待办（文本划线置灰），右侧提供“进入会议 / 开始笔试”直达按钮。
*   **后端支撑：** 
    *   `GET /api/v1/tasks?month=YYYY-MM`：按月预加载任务数据，减轻 DB 压力。
    *   `PATCH /api/v1/tasks/{id}/status`：更新待办勾选状态。

### 3.3 核心组件：多态底部工作台 (Polymorphic Bottom Sheet)
*   **前端交互：** 
    *   摒弃传统的右侧抽屉，采用从底部滑出的沉浸式大面板。
    *   **阅读/编辑双态切换：** 点击“编辑”按钮，静态文本无缝切换为极简的 Input/Select 框。
    *   **级联防呆逻辑：** 仅当阶段选择“面试中”时，才渐进式展示“面试轮次”下拉菜单。
*   **后端支撑：** 
    *   `PUT /api/v1/jobs/{id}`：一键保存抽屉内修改的所有字段，并通过 Webhook 触发看板重绘。

## 四、 🌟 核心壁垒：三大 AI Agent 工作流

这是本产品超越传统工具，实现智能化 (Copilot) 的核心业务流。

### 🤖 工作流一：智能岗位发现与推送 (Smart Job Discovery)
*   **业务目标：** 替代用户去各大厂官网刷岗位的重复劳动。
*   **架构设计 (异步非阻塞)：**
    *   **Backend (数据采集)：** 严禁前端实时爬虫。后端设定每日凌晨执行 Cron Job，抓取目标大厂官网真实在招岗位，清洗后存入向量数据库 (Vector DB)。
    *   **Agent (意图识别 + RAG)：** 用户在右侧 AI 助手输入*“我想找北京的美团产品岗”*。Agent 将自然语言转化为 `{city: "北京", role: "产品", company: "美团"}` 进行向量检索，秒级返回精准卡片。
    *   **Action (一键入库)：** 推荐的岗位卡片附带 `[加入看板]` 按钮，点击直接调用 `POST /api/v1/jobs` 生成追踪任务。

### 🤖 工作流二：简历+JD 双向解析模拟面试 (Tailored Mock Interview)
*   **业务目标：** 缓解面试焦虑，提供高针对性的实战演练。
*   **触发路径：** 在职位详情抽屉中，点击【AI 模拟面试】。
*   **Agent 工作流逻辑：**
    1.  **Context 组合：** 系统自动拉取该职位的 `JD (岗位描述)` 和用户系统内绑定的 `Resume (PDF解析文本)`。
    2.  **Prompt 注入：** `System: 你现在是[目标公司]的面试官，请根据以下这份候选人简历，针对这个[岗位要求]，用严苛的业务视角对他进行多轮提问，必须深挖他的项目细节。`
    3.  **多轮交互：** AI 发起第一问 ➡️ 用户输入回答 ➡️ AI 评价回答并追问（保持上下文窗口）。
    4.  **复盘报告：** 结束时，Agent 生成包含“逻辑漏洞、STAR 法则建议、参考回答”的复盘报告，自动存入数据库的 `notes` 字段。

### 🤖 工作流三：NLU 语音/文本极速建档 (Natural Language Data Entry)
*   **业务目标：** 解决手机端或繁忙时，手工填表（公司、时间、链接等 10 几个字段）极其痛苦的问题。
*   **Agent 工作流逻辑：**
    1.  用户在 AI 助手输入框发送邮箱或者岗位截图，图中包含信息如面试时间，岗位信息，会议链接，截止日期，面试轮次等信息，ai自动解析提取关键文字。
    2.  **大模型实体抽取 (Entity Extraction)：** Agent 解析出 JSON 数据：`{"company": "滴滴", "stage": "面试中", "round": "技术二面", "time": "明天 14:00", "url": "http://xxx"}`。如果还有岗位信息也同时解析出来，放在求职卡片的岗位描述上。
    3.  **二次确认与执行：** AI 弹出结构化的预览卡片询问“是否为您建立日程？”。用户点击确认后，同时触发 `POST /api/jobs` (建立看板卡片) 和 `POST /api/tasks` (建立日历待办)。

## 五、 数据字典规范 (Schema - 核心)

为了支撑上述业务，后端数据库必须建立以下关联表：

**1. `Table: Jobs` (求职看板表)**
*   `job_id` (PK)
*   `company_name` (String)
*   `position_title` (String)
*   `current_stage` (Enum: 待投递, 面试中, Offer...)
*   `interview_round` (Enum: 初面, HR面...)
*   `job_description` (Text) 
*   `resume_version_url` (String)

**2. `Table: Tasks` (日程待办表)**
*   `task_id` (PK)
*   `linked_job_id` (FK -> Jobs.job_id, nullable)
*   `task_title` (String)
*   `task_date` (Date)
*   `action_url` (String) - 用于存放会议或笔试链接
*   `is_completed` (Boolean)

### 💡 架构设计说明
本项目在设计之初，不仅聚焦于高保真的前端交互（解决了 Drag & Drop 动画冲突、组件状态解耦等工程痛点），更在产品架构上完成了从**“记录工具 (Tool)”**向**“智能副驾 (Copilot)”**的底层设计。
所规划的三大 Agent 工作流均充分考虑了真实商业环境中的反爬风控策略、大模型 Latency（延迟体验）以及数据隔离问题，是一套具备极高落地可行性的全栈级产品方案。
